Accessible Button

WCAG Guidelines for Buttons

Key WCAG Requirements for Buttons

  • Perceivable: Buttons must be clearly visible with sufficient color contrast (WCAG 1.4.3)
  • Operable: Buttons must be keyboard accessible (WCAG 2.1.1)
  • Focus Indicators: Buttons must have visible focus indicators (WCAG 2.4.7)
  • Name, Role, Value: Buttons must have clear, descriptive text (WCAG 4.1.2)

Implementation Notes

This button example includes:

  • Proper semantic HTML using the <button> element
  • High contrast colors for visibility
  • Clear focus states for keyboard navigation
  • Descriptive text that indicates the purpose
  • Appropriate padding for easy touch target (44px × 44px minimum, per WCAG 2.5.5)

Additional Guidelines for Icon Buttons

  • Text Alternative (WCAG 1.1.1): Icon-only buttons must have a text alternative via aria-label or similar
  • Decorative Images (WCAG 1.1.1): Decorative elements should be hidden from screen readers with aria-hidden="true"
  • Touch Target Size (WCAG 2.5.5): Icon buttons must have sufficient size for touch interaction

Implementation Notes for Icon Buttons

The icon button examples include:

  • Proper semantic <button> element
  • Accessible name provided with aria-label
  • SVG icons marked as decorative with aria-hidden="true"
  • Sufficient touch target size (44px × 44px)
  • Visual feedback on hover, focus, and active states

WCAG Guidelines for Dropdowns/Selects

Key WCAG Requirements for Dropdowns

  • Keyboard Accessibility (WCAG 2.1.1): Must be fully operable via keyboard
  • Focus Management (WCAG 2.4.3): Focus must move in a logical sequence
  • Focus Visibility (WCAG 2.4.7): Focus indicators must be visible
  • Name, Role, Value (WCAG 4.1.2): Proper ARIA roles and states for custom dropdowns
  • Status Messages (WCAG 4.1.3): Changes in state must be announced to screen readers

Implementation Notes

This example includes two approaches:

1. Native Select Element

  • Uses the browser's built-in <select> element which has excellent accessibility support
  • Properly labeled with <label> and for attribute
  • Additional context provided with aria-describedby
  • Minimal styling to maintain accessibility while improving appearance

2. Custom Dropdown

  • Implements the ARIA design pattern for a Listbox
  • Uses appropriate ARIA roles: listbox and option
  • Manages focus appropriately when opened/closed
  • Supports keyboard navigation (arrow keys, Enter, Escape)
  • Announces selection changes to screen readers

Note: When possible, use the native select element as it has the best built-in accessibility. Custom dropdowns should only be used when additional styling or functionality is required, and must be carefully implemented to maintain accessibility.

WCAG Guidelines for Progress Bars

Key WCAG Requirements for Progress Bars

  • Name, Role, Value (WCAG 4.1.2): Progress bars must have proper roles and states
  • Status Messages (WCAG 4.1.3): Changes in progress must be announced to screen readers
  • Non-text Contrast (WCAG 1.4.11): Visual indicators must have sufficient contrast
  • Adaptable (WCAG 1.3.1): Information must be available programmatically

Implementation Notes

This example includes two types of progress bars:

1. Determinate Progress Bar

  • Uses role="progressbar" to identify the element as a progress indicator
  • Provides current value with aria-valuenow
  • Defines range with aria-valuemin and aria-valuemax
  • Associates with a label using aria-labelledby
  • Includes text representation of the progress value
  • Updates ARIA attributes when the value changes

2. Indeterminate Progress Bar

  • Uses role="progressbar" but omits value attributes
  • Uses aria-busy="true" to indicate loading state
  • Provides a label with aria-label
  • Uses animation to indicate ongoing activity

Both progress bars maintain sufficient color contrast and provide both visual and programmatic indication of their state.

WCAG Guidelines for Data Tables

Key WCAG Requirements for Tables

  • Info and Relationships (WCAG 1.3.1): Use proper table markup with headers and cells
  • Meaningful Sequence (WCAG 1.3.2): Ensure logical reading order
  • Identify Purpose (WCAG 1.3.6): Clearly identify the table's purpose
  • Resize Text (WCAG 1.4.4): Table must be usable when text is resized
  • Keyboard Accessible (WCAG 2.1.1): Table interactions must work with keyboard

Implementation Notes

This data table example includes:

  • Proper semantic structure with <table>, <thead>, <tbody>, <th>, and <td> elements
  • Appropriate scope attributes on header cells (scope="col" for column headers, scope="row" for row headers)
  • Table caption for screen readers using <caption>
  • Associated header text using aria-labelledby
  • Additional description using aria-describedby
  • Responsive design that allows horizontal scrolling on small screens
  • Sufficient color contrast for all text
  • Interactive features (sorting, filtering) that maintain accessibility
  • Zebra striping for easier visual tracking across rows

Screen Reader Considerations

Screen readers will announce:

  • The table's title and description
  • The number of rows and columns
  • Header information as users navigate through cells
  • Changes to the table when sorting or filtering

WCAG Guidelines for Tabs

Key WCAG Requirements for Tabs

  • Keyboard Accessibility (WCAG 2.1.1): Tabs must be fully operable via keyboard
  • Focus Management (WCAG 2.4.3): Focus must move in a logical sequence
  • Focus Visibility (WCAG 2.4.7): Focus indicators must be visible
  • Name, Role, Value (WCAG 4.1.2): Proper ARIA roles and states for tabs
  • Info and Relationships (WCAG 1.3.1): Relationship between tabs and panels must be programmatically determined

Implementation Notes

This tabs component follows the WAI-ARIA Tabs Pattern and includes:

  • Proper ARIA roles: tablist, tab, and tabpanel
  • Appropriate ARIA states: aria-selected, aria-controls, and aria-labelledby
  • Keyboard navigation:
    • Tab/Shift+Tab: Move focus into and out of the tablist
    • Arrow keys: Navigate between tabs when focus is in the tablist
    • Space/Enter: Activate the tab that has focus
  • Focus management that follows the roving tabindex pattern
  • Hidden panels use the hidden attribute to ensure they're properly hidden from all users
  • Visible focus indicators for keyboard users
  • Semantic heading structure within each panel

WCAG Guidelines for Switches

Key WCAG Requirements for Switches

  • Name, Role, Value (WCAG 4.1.2): Switches must have proper roles and states
  • Keyboard Accessibility (WCAG 2.1.1): Switches must be operable via keyboard
  • Focus Indicators (WCAG 2.4.7): Switches must have visible focus indicators
  • Labels (WCAG 3.3.2): Switches must have clear, descriptive labels
  • Non-text Contrast (WCAG 1.4.11): Visual state indicators must have sufficient contrast

Implementation Notes

This switch component includes:

  • Proper semantic structure using role="switch"
  • Current state indicated with aria-checked
  • Associated labels using aria-labelledby
  • Additional descriptions using aria-describedby where needed
  • Support for disabled state
  • Keyboard accessibility (Space/Enter to toggle)
  • High contrast visual indicators
  • Animated transitions that respect reduced motion preferences
  • Touch-friendly size for mobile users

Why Use a Switch Instead of a Checkbox?

Switches are best used for:

  • Immediate actions or settings that take effect right away
  • Binary on/off states (rather than selected/unselected)
  • Settings that toggle features or modes

Checkboxes are better for:

  • Selecting items from a list
  • Options that will be submitted as part of a form
  • When multiple related options can be selected

WCAG Guidelines for Sliders

Key WCAG Requirements for Sliders

  • Keyboard Accessibility (WCAG 2.1.1): Sliders must be operable via keyboard (arrow keys, Home/End)
  • Focus Indicators (WCAG 2.4.7): Sliders must have visible focus indicators
  • Labels (WCAG 3.3.2): Sliders must have clear, descriptive labels
  • Name, Role, Value (WCAG 4.1.2): Slider state and value must be programmatically determined
  • Non-text Contrast (WCAG 1.4.11): Slider controls must have sufficient contrast

Implementation Notes

This slider component includes:

  • Native <input type="range"> for maximum compatibility
  • Proper labeling with associated <label> elements
  • ARIA attributes to enhance screen reader experience:
    • aria-valuemin, aria-valuemax, aria-valuenow
    • aria-valuetext for human-readable value descriptions
  • Visual value display that updates in real-time
  • Enhanced styling with custom thumb and track
  • Tick marks for reference points on applicable sliders
  • Increment/decrement buttons for fine control
  • Appropriate step values for different use cases

Keyboard Navigation

  • Left/Down Arrow: Decrease value by one step
  • Right/Up Arrow: Increase value by one step
  • Home: Set to minimum value
  • End: Set to maximum value
  • Page Up: Increase by larger step (10% of range)
  • Page Down: Decrease by larger step (10% of range)

WCAG Guidelines for Avatars

Key WCAG Requirements for Avatars

  • Text Alternatives (WCAG 1.1.1): Images must have appropriate alt text
  • Color Contrast (WCAG 1.4.3): Text and status indicators must have sufficient contrast
  • Keyboard Accessibility (WCAG 2.1.1): Interactive avatars must be keyboard accessible
  • Focus Indicators (WCAG 2.4.7): Interactive avatars must have visible focus indicators
  • Name, Role, Value (WCAG 4.1.2): Interactive avatars must have proper roles and accessible names

Implementation Notes

This avatar component includes several variations:

  • Image Avatar: Uses an actual photo with appropriate alt text
  • Initials Avatar: Uses text initials with the full name available to screen readers
  • Icon Avatar: Uses a generic icon for users without photos
  • Interactive Avatar: Clickable avatar with proper button semantics
  • Group Avatar: Represents multiple users with appropriate labeling

Accessibility Considerations

  • Alt Text Strategy:
    • For standalone avatars, use descriptive alt text: "Profile photo of [Name]"
    • For decorative avatars where the name is already visible nearby, use empty alt text
    • For purely decorative elements, use aria-hidden="true"
  • Status Indicators: Visual status indicators (online/offline) are supplemented with text
  • Interactive Elements: Clickable avatars use proper button semantics
  • Focus States: Clear visual indication when interactive avatars have keyboard focus
  • Color Independence: Status is not conveyed by color alone

WCAG Guidelines for Dialogs

Key WCAG Requirements for Dialogs

  • Focus Management (WCAG 2.4.3): Focus must move to the dialog when opened
  • Focus Containment (WCAG 2.1.2): Focus must be trapped within the dialog while open
  • Keyboard Accessibility (WCAG 2.1.1): All dialog controls must be keyboard accessible
  • Name, Role, Value (WCAG 4.1.2): Dialogs must have proper ARIA roles and labels
  • Escape Key (WCAG 2.1.4): Dialogs should be dismissible with the Escape key
  • Modal Behavior (WCAG 1.4.13): Content behind the dialog should be inaccessible

Implementation Notes

This dialog component includes:

  • Standard Dialog: For general content, forms, and settings
  • Alert Dialog: For important notifications requiring user confirmation
  • Proper ARIA roles: role="dialog" or role="alertdialog"
  • Proper labeling with aria-labelledby and aria-describedby
  • Modal behavior with aria-modal="true"
  • Focus management that:
    • Moves focus to the dialog when opened
    • Traps focus within the dialog while open
    • Returns focus to the triggering element when closed
  • Keyboard support:
    • Escape key to close
    • Tab/Shift+Tab to navigate within the dialog
  • Visual design that:
    • Clearly separates the dialog from background content
    • Provides clear visual hierarchy
    • Includes visible close mechanisms

Dialog vs. Alert Dialog

Standard Dialog (role="dialog"):

  • Used for non-critical interactions
  • May contain complex forms or content
  • Focus typically starts on the first focusable element

Alert Dialog (role="alertdialog"):

  • Used for critical information requiring immediate attention
  • Typically contains a warning or error message
  • Requires user acknowledgment or decision
  • Focus typically starts on the primary action button

WCAG Guidelines for Tooltips

Key WCAG Requirements for Tooltips

  • Text Alternatives (WCAG 1.1.1): Icon buttons with tooltips must have accessible names
  • Keyboard Accessibility (WCAG 2.1.1): Tooltips must be accessible via keyboard
  • Focus Management (WCAG 2.4.3): Tooltips should appear on focus, not just hover
  • Content on Hover (WCAG 1.4.13): Tooltips must:
    • Be dismissible without moving pointer or keyboard focus
    • Remain visible until dismissed, hover/focus is removed, or content is no longer valid
    • Persist when the user moves the pointer from the trigger to the tooltip content
  • Name, Role, Value (WCAG 4.1.2): Tooltips must have proper ARIA roles and relationships

Implementation Notes

This tooltip component includes:

  • Proper ARIA attributes: role="tooltip" and aria-describedby
  • Keyboard accessibility:
    • Tooltips appear on focus, not just hover
    • Non-interactive elements with tooltips are made focusable with tabindex="0"
    • Tooltips can be dismissed with the Escape key
  • Visual design:
    • High contrast text for readability
    • Sufficient padding and spacing
    • Directional arrow to indicate the relationship to the trigger
  • Behavior:
    • Tooltips remain visible when moving from trigger to tooltip content
    • Tooltips can be persistent when needed for complex information
    • Tooltips are positioned to avoid obscuring important content

When to Use Tooltips

Tooltips are best used for:

  • Providing additional context for buttons or icons
  • Explaining form fields or input requirements
  • Clarifying terminology or features
  • Providing hints without cluttering the interface

Tooltips should not be used for:

  • Essential information required to complete a task
  • Complex content that requires interaction
  • Critical warnings or error messages
  • Information that all users need to see

WCAG Guidelines for Badges and Chips

Key WCAG Requirements for Badges and Chips

  • Color Contrast (WCAG 1.4.3): Badge text and background must have sufficient contrast
  • Color Independence (WCAG 1.4.1): Information must not be conveyed by color alone
  • Keyboard Accessibility (WCAG 2.1.1): Interactive chips must be keyboard accessible
  • Focus Indicators (WCAG 2.4.7): Interactive chips must have visible focus indicators
  • Name, Role, Value (WCAG 4.1.2): Chips must have proper roles and accessible names
  • Touch Target Size (WCAG 2.5.5): Interactive elements must have sufficient size

Implementation Notes

This badge/chip component includes several variations:

  • Status Badges: Simple indicators for status information
  • Notification Badges: Numeric indicators for unread or new items
  • Filter Chips: Interactive elements for filtering content
  • Input Chips: Interactive elements for managing tags or selections

Accessibility Considerations

  • Status Badges:
    • Use semantic HTML to ensure badges are properly announced
    • Ensure sufficient color contrast between text and background
    • Don't rely on color alone to convey status (use text)
  • Notification Badges:
    • Include the notification count in the accessible name of the associated button
    • Use aria-hidden="true" on the visual badge since the information is already in the button's accessible name
  • Interactive Chips:
    • Use appropriate ARIA roles (role="checkbox" for filter chips)
    • Ensure keyboard accessibility with proper focus management
    • Provide clear visual indication of focus and selection states
    • Ensure remove buttons have descriptive accessible names
  • Input Chips:
    • Ensure proper keyboard navigation between chips and input field
    • Provide clear instructions for adding and removing chips
    • Announce changes to the chip collection to screen reader users
